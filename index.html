A Two-Part Approach to Fixing XSS

  1. Validate Input
  2. Sanitize Output

Validation

Best case: Compare against an Allow List of known-good values

e.g.


  var HANDEDNESS = ['Lefty','Righty','Ambidexterous','Other'];

but

Human names don't fit into a convenient list :(

Sanitization

(a.k.a. filtering, normalizing, or escaping)

Prevent user-controlled data from breaking out of its context.

E.g. from HTML into <script>

Fixed example (EJS)

Change ...



  <h1>Hello <%- user.name %>, welcome to <%- site.name %></h1>

... to ...



  <h1>Hello <%= user.name %>, welcome to <%- site.name %></h1>

Where <%= %> is an Escaping operator for a Template Slot.

Is XSS Just for Templates?

No

There's more than just HTML context

jQuery XSS

Vulnerable:


  jQuery('username').html(user.name)

Safe: (.text())


  jQuery('username').text(user.name)

Plain JavaScript XSS

Vulnerable:


  var username = document.getElementById('username');
  username.innerHTML = user.name;

Safe: document.createTextNode()


  var username = document.getElementById('username');
  username.appendChild(document.createTextNode(username));

CSRF

Cross-Site Request Forgery

CSRF exploits the fact that you are logged-in to some other site.

For example,

  • You're logged into https://example.com
  • You go to visit http://evil.com

evil.com has the following markup:


  <title>Welcome to Evil.com</title>
  <script src="https://example.com/api/inviteAdmin?email=hacker@evil.com">

Even though you're visiting evil.com,

you're still authenticated with example.com!

How do we fix this?

In Human terms:

Assert that the user intended to do this action.

E.g. the user was on my website ...

... then, they clicked submit on a form ...

... therefore, this isn't a Cross-Site Forgery

In Technical terms:

"The user was on my website ..."

=

Provide the user a unique, secret Anti-CSRF token in addition to their login-cookie.

"... then they clicked submit on a form ..."

=

Actions that change application state should be POST/PUT/PATCH/DELETE (consistent with the REST Architectural Style)

"... therefore, this isn't a Cross-Site Forgery"

=

Validate the Anti-CSRF token, which MUST be kept secret from attackers

Fixing example.com

Assume it's running a simple Express 3.x node.js server with EJS templates.

Express Routes


  app.get('/api/inviteAdmin', handlerFn);

... change this to ...


  app.post('/api/inviteAdmin', handlerFn);

Connect Anti-CSRF middleware

www.senchalabs.org/connect/csrf.html 

  app.use(connect.session());
  app.use(connect.csrf());

... then to access the token ...


  var token = req.csrfToken();
  res.render('template', { _csrf: token });

(Connect's CSRF middleware is also BREACH attack resistant)

Change the EJS template


  <form method="GET" action="/api/inviteAdmin">
    <input type="email" name="email">

... change to use POST and consume anti-CSRF token ...


  <form method="POST" action="/api/inviteAdmin">
    <input type="hidden" name="_csrf" value="<%- _csrf %>">
    <input type="email" name="email">
What if I have to use a GET?

Forced-use of GET

Remember the principal idea: verify that the Human intended to do this

Idea: If it's a link to be clicked, convert the link to invoke a form-POST instead!

If not, it may be possible to ask the user for confirmation some how.

CSRF summary

You too can prevent forest CSRF fires!

  1. Verify the user indended to do this action.
  2. Be a good REST citizen: GET should not change application state
